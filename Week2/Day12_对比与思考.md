## 一、手写 vs 框架：全方位对比（15 分钟）

这周你已经有了两个版本的 Agent 实现——Week 1 的原生 SDK 手写版和 Day 11 的 LangChain 版。现在从工程视角做一次系统性对比。

### 核心概念

| 概念 | 说明 | 要点 |
|------|------|------|
| **控制力（Control）** | 对 Agent 行为细节的掌控程度 | 手写 > 框架，但维护成本也随之增加 |
| **开发效率（Velocity）** | 从零到可用产品的时间 | 框架 > 手写，但前提是场景匹配 |
| **可调试性（Debuggability）** | 出问题时定位和修复的难度 | 手写代码透明可控，框架有"黑盒"风险 |
| **可扩展性（Extensibility）** | 添加新功能的难度 | 框架有插件生态优势，手写更灵活 |

### 六维对比表

| 维度 | 手写 Agent（原生 SDK） | 框架 Agent（LangChain） |
|------|----------------------|------------------------|
| **学习曲线** | 低——只需 API 文档 | 中——需学习框架抽象和 API |
| **开发速度** | 慢——所有胶水代码自己写 | 快——开箱即用 |
| **灵活性** | ⭐⭐⭐⭐⭐ 完全控制流程 | ⭐⭐⭐ 受框架抽象约束 |
| **可维护性** | 中——代码量大，但逻辑透明 | 高——标准化组件，团队协作好 |
| **调试难度** | 低——所有代码一目了然 | 中——需理解框架内部执行流程 |
| **社区资源** | 少——全靠自己 | 多——文档、教程、插件丰富 |

> **重要认知转变**：没有"最好"的方案，只有"最适合"的方案。关键是建立选型决策框架，而不是盲目追框架或抗拒框架。

### Step 3：理解框架的"隐藏成本"

使用框架并非零成本，需要意识到以下"隐藏成本"：

| 隐藏成本 | 具体表现 | 应对策略 |
|----------|---------|---------|
| **版本升级** | LangChain 迭代速度极快，API 频繁变动 | 锁定版本，定期评估升级必要性 |
| **调试黑盒** | 报错信息可能指向框架内部，而非你的代码 | 善用 `verbose=True` 和 LangSmith |
| **性能开销** | 框架增加了抽象层，多了序列化/反序列化 | 性能敏感场景考虑手写 |
| **过度依赖** | 团队只会用框架，不理解底层原理 | 先学手写，再用框架（你已经做到了） |

## 三、LlamaIndex 初探（15 分钟）

### LlamaIndex vs LangChain：定位差异

很多人会问"LlamaIndex 和 LangChain 有什么区别？选哪个？" 答案是：**它们解决的核心问题不同**。

| 维度 | LangChain | LlamaIndex |
|------|-----------|-----------|
| **核心定位** | 通用 LLM 应用开发框架 | 数据索引与检索（RAG）专家 |
| **擅长场景** | Agent、Chain、多步骤流程、工具编排 | 文档索引、语义搜索、知识库问答 |
| **核心抽象** | Chain / Agent / Memory | Index / Retriever / QueryEngine |
| **工具生态** | 丰富——搜索、数据库、API 等 | 聚焦数据连接——PDF、数据库、API 数据源 |
| **学习曲线** | 中等——概念多，但文档完善 | 较低——专注 RAG 场景，API 简洁 |
| **组合能力** | ⭐⭐⭐⭐⭐ LCEL 管道语法灵活 | ⭐⭐⭐ 聚焦检索流程 |

### 思考：什么时候选哪个？

```python
"""
选型速查表（写在注释中方便以后查阅）：

1. 纯 RAG/知识库问答 → LlamaIndex（更专业、更简洁）
2. Agent + 工具调用 → LangChain（工具生态更丰富）
3. RAG + Agent 结合 → LangChain + LlamaIndex（可以混用）
4. 高度定制化流程 → 手写 + 按需引入框架组件
5. 快速 MVP 验证 → LangChain（社区资源最多）

实际生产中，很多项目会同时用两个框架：
- LlamaIndex 负责 RAG 检索部分（索引、分块、检索、重排序）
- LangChain 负责 Agent 编排部分（工具调用、对话管理、流程控制）
"""
```

### LlamaIndex 快速体验（选读）

```powershell
# 如果想提前体验，可以安装
pip install llama-index llama-index-llms-openai
```

```python
# ── LlamaIndex 快速体验（可选）────────────────────────────
# 以下代码展示 LlamaIndex 的核心风格，无需运行
# 第 4 周 RAG 学习时会深入使用

from llama_index.core import VectorStoreIndex, SimpleDirectoryReader

# 三行代码搭建一个文档问答系统
documents = SimpleDirectoryReader("./my_docs").load_data()  # 加载文档
index = VectorStoreIndex.from_documents(documents)          # 构建索引
query_engine = index.as_query_engine()                      # 创建查询引擎

# 提问
response = query_engine.query("这个项目的架构是什么？")
print(response)
```

> **观察要点**：对比 LlamaIndex 的三行代码 vs 如果用 LangChain 实现同样功能需要的代码量。在 RAG 场景下，LlamaIndex 的抽象层次更高、更简洁。

## 四、关键要点总结

| # | 要点 | 说明 |
|---|------|------|
| 1 | 选型的核心是"场景匹配" | 没有最好的方案，只有最合适的 |
| 2 | 先理解原理，再用框架 | 你已经具备这个基础了（Week 1 手写经验） |
| 3 | LangChain = 通用 Agent 框架 | 擅长工具编排、多步骤流程 |
| 4 | LlamaIndex = RAG 专家 | 擅长数据索引和检索，两者可混用 |
| 5 | 框架有"隐藏成本" | 版本迭代、调试黑盒、性能开销、过度依赖 |
