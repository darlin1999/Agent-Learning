
## 三、Step 1：实现 `tools.py`（40 分钟）

这是你今天的第一个编码任务。`tools.py` 需要包含三样东西：

### 3.1 工具函数

你需要实现至少两个工具函数：

**工具 A：获取当前时间**  
- 函数名建议：`get_current_time`  
- 返回当前的日期、时间、星期几  
- 返回值格式：JSON 字符串（和 Day 4 一致）  

**工具 B：数学计算器**  
- 函数名建议：`calculate`  
- 接收一个数学表达式字符串，返回计算结果  
- ⚠️ 安全要求：**必须**做输入过滤，不能直接 `eval()` 任意字符串  
- 提示：回忆 Day 4 中的白名单过滤方式

> 💡 这两个函数你在 Day 4 已经写过了。但请**不要复制粘贴**，而是凭理解重新写一遍。原因：
> - 重写能暴露你真正不确定的地方
> - 你可能会想到更好的写法（比如更清晰的错误提示）
> - 实际开发中你也需要在不同上下文中重新实现类似逻辑

### 3.2 工具描述列表（`tools`）

为每个工具函数编写 JSON Schema 格式的描述，组成 `tools` 列表。回忆 Day 4 的结构：

```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "...",
            "description": "...",    # 这段描述决定 LLM 何时选用此工具
            "parameters": {
                "type": "object",
                "properties": { ... },
                "required": [ ... ],
            },
        },
    },
    # ... 更多工具
]
```

> ⚠️ 回忆 Day 4 的关键规律：**`description` 的质量直接决定 LLM 能否正确判断何时使用该工具**。别写"一个工具"这种模糊描述。

### 3.3 工具调度映射

提供一种方式，让 `main.py` 能根据函数名字符串找到对应的函数并执行。Day 4 中用了一个字典：

```python
TOOL_FUNCTIONS = {
    "函数名": 函数引用,
    ...
}
```

你也可以选择其他方式（比如导出一个 `execute_tool(name, args)` 函数），只要 `main.py` 能方便地调用就行。

写完 `tools.py` 后，在脑中（或终端）验证：

## 四、Step 2：实现 `main.py` 的核心——对话循环（60 分钟）

这是今天最核心的部分。`main.py` 要实现一个**多轮对话循环**，每一轮的逻辑如下：

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  1. 获取用户输入                                          │
│          │                                               │
│          ▼                                               │
│  2. 追加到 messages 列表                                  │
│          │                                               │
│          ▼                                               │
│  3. 调用 LLM API（带 tools 参数）                         │
│          │                                               │
│          ▼                                               │
│  4. 检查返回：LLM 要调工具，还是直接回答？                  │
│          │                    │                           │
│     有 tool_calls        无 tool_calls                    │
│          │                    │                           │
│          ▼                    ▼                           │
│  5a. 执行工具函数        5b. 打印回复                      │
│  5a-2. 结果追加到 messages   5b-2. 回复追加到 messages      │
│  5a-3. 回到步骤 3 ───┐       │                            │
│          ▲            │       │                           │
│          └────────────┘       │                           │
│                               ▼                          │
│                        6. 等待下一轮用户输入               │
│                               │                          │
│                               └────── 回到步骤 1          │
└──────────────────────────────────────────────────────────┘
```

### 4.1 需要你自己解决的核心问题

以下问题在 Day 2-4 的学习中都有答案，但今天需要你**自己组装**：

**问题 A：`messages` 列表如何初始化？**  
- 提示：需要一条 `system` 消息来设定 ChatBot 的角色和行为
- 思考：这条 system prompt 应该写什么？你在 Day 3 学到的技巧可以用上

**问题 B：用户输入用什么 Python 函数获取？**  
- 提示：Python 内置函数，不需要导包
- 思考：用户输入 `quit` 时如何退出循环？输入空字符串时要不要跳过？

**问题 C：LLM 返回 `tool_calls` 时，消息怎么追加？**  
- 提示：Day 4 第六章"核心代码逐行解析"讲过，需要追加**两种**消息
- ⚠️ 这是最容易出错的地方。忘记追加 `assistant_message` 或者 `tool_call_id` 写错，都会导致 API 报错

**问题 D：如何支持 LLM 连续多次调用工具？**  
- 这是 Day 4 "挑战 C"提到但未给答案的问题
- 提示：步骤 5a-3 回到步骤 3，意味着你需要一个**内层循环**
- 关键判断条件：`finish_reason` 是 `"tool_calls"` 还是 `"stop"`

**问题 E：如何让对话有记忆？**  
- 这个其实最简单——`messages` 列表在用户输入之间不要清空就行
- 但要注意：随着对话越来越长，messages 会越来越大，可能超出模型的上下文窗口。今天先不处理这个问题（第 3 周会学 Memory 机制），但你应该意识到它的存在

### 4.3 关键提醒

1. **`messages.append(assistant_message)` 不要忘**——当 LLM 返回 `tool_calls` 时，必须把整个 assistant message 对象追加进 messages，才能在下一次请求中带上
2. **`tool_call_id` 必须匹配**——每个 `role: "tool"` 消息的 `tool_call_id` 必须对应其回应的 `tool_call.id`
3. **`json.loads()` 解析 arguments**——Day 4 踩过的坑，arguments 是字符串不是字典
4. **对话循环 vs 工具循环**——对话循环是外层的 `while True`（用户一轮轮输入），工具循环是内层的（LLM 可能连续请求多个工具）。两个循环嵌套，不要混淆

### 5.3 常见问题排查

| 报错信息 / 现象 | 原因 | 修复方向 |
|----------------|------|---------|
| `400 Bad Request` / messages 格式错误 | tool_calls 后未正确追加 assistant message 或 tool result | 检查 messages 追加顺序：assistant → tool |
| `tool_call_id` 不匹配 | 复制了错误的 id，或遗漏了某个 tool_call 的结果 | 确保每个 tool_call 都有对应的 tool result |
| `json.JSONDecodeError` | `tool_call.function.arguments` 未做 `json.loads()` | 解析后再传参 |
| 对话没有记忆（"刚才"无效） | messages 在循环中被重置了 | messages 应在 `while True` 外面初始化 |
| 工具调用后 LLM 又调用一次工具（无限循环） | 内层工具循环没有正确的退出条件 | 检查 `finish_reason` 或 `tool_calls` 是否为空 |

## 八、关键规律总结

| # | 规律 | 今日体会 |
|---|------|---------|
| 1 | **对话历史就是 messages 列表**：多轮对话的"记忆"完全靠维护一个不断增长的 messages 数组实现，没有任何魔法 | 核心对话循环 |
| 2 | **工具调用是中间步骤，不是最终回答**：LLM 返回 tool_calls 时 content 通常为空，你必须执行工具并把结果喂回去，LLM 才会给出最终文本 | 工具调度逻辑 |
| 3 | **模块化从第一个项目就开始**：`tools.py` 和 `main.py` 的分离让加新工具变得无损——这就是传统软件工程中"关注点分离"的老原则在 Agent 领域的体现 | 项目结构 |
| 4 | **先让它跑起来，再优化**：不要一开始就追求完美的错误处理和漂亮的输出，先实现核心流程，再逐步打磨 | 整体开发节奏 |
| 5 | **调试 Agent 要看完整的 messages 链**：出问题时 `print(json.dumps(messages, ensure_ascii=False, indent=2))` 是最有效的调试手段 | 调试过程 |
