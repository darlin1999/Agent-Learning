> **目标**：使用原生 `openai` SDK（对接 DeepSeek）独立实现一个命令行 ChatBot，把 Day 3 的 Prompt 设计、Day 4 的 Function Calling、Day 5 的工程规范串起来，做出一个可演示、可继续扩展的小项目。  
> **时间**：3 小时  
> **前置条件**：已完成 Day 2-5，虚拟环境可用，`.env` 可正常读取 `DEEPSEEK_API_KEY`，已能运行 Day 4 的工具调用示例。  
> **完成方式**：这是一份**带引导的小作业**。文档会给你明确的实现路径、接口约束、调试顺序和验收标准，但**不会直接给出关键成品代码**。必须由你自己完成核心实现。

## 零、本次作业定位

今天不是继续学新概念，而是做一次**受控实战整合**。

你要做的不是一个“看起来能跑”的一次性脚本，而是一个最小可用、结构清晰的 ChatBot 原型。它至少要具备这 3 类能力：

1. **普通对话**：用户问一个普通问题，它能直接回答
2. **工具调用**：用户问时间、计算题等问题时，它会调用本地工具，而不是胡编
3. **多轮运行**：程序不只跑一轮，而是能持续对话，直到用户输入退出命令

> 这次作业的重点不是“代码量”，而是你是否真正理解：
> - 什么时候该让模型直接回答
> - 什么时候该通过 `tools` 触发工具调用
> - 工具结果如何再回传给模型
> - 如何把这一套流程组织成一个可复用的小程序

## 五、程序整体结构先想清楚

在开始写代码前，先用一句话描述你的程序结构：

> “一个循环读取用户输入的命令行 ChatBot。它把对话历史发给模型；如果模型要求调用工具，就由本地 Python 函数执行，再把结果回传给模型生成最终答复。”

### 你至少需要这 5 个组成部分

| 组成部分 | 放在哪 | 作用 |
|---------|-------|------|
| 环境变量加载 | `main.py` | 读取 API Key |
| 客户端初始化 | `main.py` | 创建 DeepSeek 对应的 `OpenAI` client |
| 本地工具函数 | `tools.py` | 真正执行时间/计算等逻辑 |
| 工具描述声明 | `main.py` 或独立模块 | 告诉模型“可调用什么工具” |
| 对话主循环 | `main.py` | 持续收用户输入、请求模型、输出结果 |

### 推荐的执行流

```text
程序启动
  ↓
加载 .env
  ↓
初始化 client
  ↓
准备 tools 描述
  ↓
进入 while 循环
  ↓
读取用户输入
  ↓
发送第一轮请求
  ↓
是否有 tool_calls？
  ├─ 否：直接输出模型回复
  └─ 是：执行本地工具
           ↓
         把工具结果追加进 messages
           ↓
         发送第二轮请求
           ↓
         输出最终回复
  ↓
继续下一轮
```

## 六、分阶段完成，不要一口气写完

这次作业最容易失败的方式，是从第一分钟开始就试图把所有代码一次性写全。正确做法是**分 4 个阶段逐步打通**。

### 阶段 1：先做一个“不带工具的聊天外壳”

先只实现下面这条最简单的链路：

```text
用户输入 -> 调用模型 -> 打印回复
```

#### 阶段目标

- 能从终端读取输入
- 能正常请求 `deepseek-chat`
- 能打印模型返回文本
- 能识别退出命令

#### 你自己必须写出的内容

- `.env` 加载
- `OpenAI(...)` 客户端初始化
- `messages` 的基本结构
- `while True` 循环

#### 提示，但不直接给答案

- 你已经在 Day 3、Day 4 里见过 `client.chat.completions.create(...)`
- `messages` 至少应包含 `system` 和 `user`
- 首先别管历史对话，先做“单轮版”也可以

#### 自测问题

如果用户输入“你是谁”，你的程序能不能在不使用工具的前提下直接回答？

---

### 阶段 2：只打通一个工具链路

不要一开始写两个工具。先只打通 `get_current_time` 一条链路。

#### 目标

让程序能正确处理下面这类问题：

- 现在几点？
- 今天星期几？
- 帮我看看当前时间

#### 你需要自己完成的关键点

1. 在 `tools.py` 中定义时间工具函数
2. 在请求里加入 `tools` 参数
3. 判断模型返回中是否存在 `tool_calls`
4. 解析函数名与参数
5. 本地执行函数
6. 将工具结果作为 `tool` 消息回传给模型
7. 发送第二轮请求并输出最终答案

#### 这里不直接给代码，但你必须清楚的接口约束

**时间工具建议函数名**：`get_current_time`

**输入参数建议**：

| 参数 | 类型 | 是否必填 | 建议说明 |
|------|------|---------|---------|
| `timezone` | `str` | 否 | 默认可用 `Asia/Shanghai` |

**返回内容建议**：

不要只返回一句自然语言，建议返回一个**结构化字符串**，至少包含：

- 时区
- 日期时间
- 星期

> 原因：结构化返回更适合后续交给模型整合，也更利于调试。

#### 第二轮回传时，你需要确保消息里至少有这些信息

- assistant 发起的工具调用信息
- tool 对应的结果
- tool 的 `tool_call_id`

#### 自测问题

如果模型已经说“我要调用工具”，你却没有把工具结果再发回去，会发生什么？

---

### 阶段 3：加入第二个工具，并完成分发逻辑

当时间工具打通后，再加 `calculate`。

#### 目标

让程序能正确处理：

- `23 * 17 等于多少？`
- `(15 + 28) / 3` 的结果是多少？

#### 这里最关键的，不是数学，而是“工具分发”

也就是：

> 当模型返回不同的 `function.name` 时，你的代码如何路由到对应 Python 函数？

#### 你需要自己设计的部分

- 是用 `if/elif` 分发，还是用字典映射分发？
- 工具函数执行失败时，你返回什么？
- 参数缺失时，你是抛异常、返回错误 JSON，还是给默认值？

#### 对 `calculate` 的最低安全要求

即使这是练习，也不要裸奔式地执行任意字符串。

至少考虑下面这件事：

- 是否限制允许的字符范围

> 这不是为了把你变成安全专家，而是为了建立一个工程习惯：  
> **凡是模型生成、用户输入、外部传入的内容，都不能无脑信任。**

#### 推荐返回格式

至少包含：

- 原始表达式
- 计算结果
- 出错时的错误信息

---

### 阶段 4：补上多轮对话体验

当两个工具都能工作后，再把程序升级成“可连续使用”的小 ChatBot。

#### 目标

- 用户可以连续问多个问题
- 每一轮都能得到清晰输出
- 支持退出命令
- 最好能保留最近几轮消息，形成上下文

#### 你需要思考的问题

1. `messages` 是每轮重建，还是持续累计？
2. 如果持续累计，是否需要限制长度？
3. 用户问“再算一下刚才那个结果乘以 2”，你的程序是否依赖上下文？

#### 本作业的建议选择

先做一个**保留上下文的基础版**，但不用做复杂裁剪。  
只要你能说清楚“为什么后续需要消息裁剪”，就已经达到本周要求。

## 七、必须掌握但不直接给代码的 4 个核心点

下面这 4 块是本次作业的真正重点。文档只给判断标准，不给成品代码。

### 1. system prompt 怎么写

你的 system prompt 至少要覆盖：

- 机器人身份
- 回答风格
- 何时应优先调用工具
- 不知道时不要编造

#### 最低要求

它至少要能约束模型做到下面两点：

1. 涉及时间、计算等确定性问题时，优先用工具
2. 工具不可用时，要明确说明限制，而不是胡编

#### 自查标准

如果你删掉 system prompt，程序大概率仍能跑，但行为会更飘。为什么？

---

### 2. `tools` 描述怎么写

你不需要死记格式，但必须知道一个工具定义里最重要的字段是什么：

| 字段 | 作用 |
|------|------|
| `name` | 告诉模型工具叫什么 |
| `description` | 告诉模型什么时候该用它 |
| `parameters` | 告诉模型参数名、类型、是否必填 |

#### 最容易出错的地方

- `description` 写得太泛，模型不容易正确判断调用时机
- `required` 漏填，导致模型可能不传关键参数
- 参数说明太弱，模型传出的值不稳定

#### 自查标准

如果一个工具的说明只写“一个函数”，模型为什么很难用好它？

---

### 3. `tool_calls` 解析怎么做

你必须明确知道：

- 模型返回的参数通常是一个 JSON 字符串
- 需要先解析，再传给 Python 函数
- 你要根据函数名决定调用哪个本地工具

#### 至少打印一次中间结果

在开发阶段，建议你临时打印这些信息：

- 返回的 `finish_reason`
- `tool_calls` 中的函数名
- `tool_calls` 中的参数原始内容

> 这是调试习惯，不是正式功能。  
> 如果中间结构看不清，后面所有错误都会变得很难查。

---

### 4. 为什么必须有“第二次请求”

这是本次作业最容易“会写但没懂”的地方。

#### 正确认知

第一次请求的作用：

- 让模型决定是否调用工具
- 告诉你调用哪个工具、传什么参数

第二次请求的作用：

- 把工具真实执行结果交给模型
- 让模型基于真实结果组织最终自然语言回答

#### 换句话说

第一次不是“得到最终答案”，而是“得到行动指令”。

## 十一、常见错误清单

如果你跑不通，优先检查这些点：

| 问题现象 | 高概率原因 |
|---------|-----------|
| 程序一启动就报 Key 相关错误 | `.env` 没读到，或变量名不一致 |
| 普通聊天能回答，但时间问题不调工具 | `tools` 没传，或工具描述写得太弱 |
| 拿到 `tool_calls` 后报错 | `arguments` 没有做 JSON 解析 |
| 工具执行了，但最终回答很怪 | 第二轮消息拼接不完整 |
| 程序只能问一轮 | 没有写循环 |
| 一输入计算题就报错 | 表达式处理过于脆弱，缺少校验 |

## 十五、今日最重要的 5 句话
